# Plan: MCP Tool CLI — stateless tool access for OpenCode

**Date**: 2026-04-02
**Status**: Draft

> **Partially superseded (2026-05-16):** Per-repo upstream scoping
> (`repos[*].proxies`, `repos[*].channels`) was removed. All upstreams are now
> available to every directory under `/workspace/repos/`. The MCP CLI shape and
> approval flow are otherwise unchanged.

## Goal

Replace OpenCode's persistent MCP connections to the proxy with a stateless CLI that lists and calls MCP tools over plain HTTP. The proxy can restart freely without breaking the agent's tool access.

## Why

OpenCode holds SSE/Streamable HTTP connections from startup. When the proxy restarts (config reload, deploy, crash), those connections break. The agent loses MCP sessions and must be restarted. A stateless CLI avoids this — each invocation is one HTTP round-trip, no session to break.

This follows the same pattern already proven with `git`/`gh` — wrapper scripts in the OpenCode container that call an HTTP service. The MCP tool CLI extends `remote-cli.mjs` to cover MCP tools.

## CLI Design

**Name**: `mcp` — short, accurate, no conflicts inside the controlled container PATH.

### Usage

Upstream and tool names are the subcommands — no `list`/`call` ceremony. Tool names pass through as-is (underscores, camelCase, etc.).

```bash
mcp                                            # list available upstreams (also: --help, -h)
mcp slack                                      # list tools on the slack upstream (also: mcp slack --help)
mcp slack postMessage --help                   # print tool description + input schema (from proxy)
mcp slack postMessage '{"channel":"C123","text":"hi"}'   # call a tool — single JSON arg → JSON response
```

### Approval CLI

Separate CLI for the approval workflow — different concern from tool discovery/execution.

```bash
approval status <action-id>     # check status (pending/approved/rejected + result)
approval list                   # list pending approvals (useful for debugging)
```

Same `remote-cli.mjs` pattern (`docker/opencode/bin/approval` wrapper), hitting `GET /approval/:id` and `GET /approvals`. The proxy's synthetic `check_approval_status` MCP tool is removed.

### Output format

Raw MCP JSON response as returned by the proxy — no pretty-printing or transformation, so it stays parseable by the agent.

### Per-repo upstream scoping

Currently, each repo's `opencode.json` controls which MCP servers it connects to. With the CLI, this moves into `config.json` — each repo declares which proxies it can access:

```json
{
  "repos": {
    "acme-app": { "channels": ["C123"], "proxies": ["slack", "atlassian"] }
  },
  "proxies": {
    "slack": { ... },
    "atlassian": { ... }
  }
}
```

**Enforcement**: The CLI sends `cwd` with every request (same as `git`/`gh` wrappers). The proxy extracts the repo name from the path (e.g. `/workspace/repos/acme-app` → `acme-app`) and filters upstreams to only those listed in that repo's `proxies` array. A repo without a `proxies` field gets no upstreams. No extra env var or header needed — repo identity is derived from the working directory.

### Environment

| Var              | Default    | Description                                     |
| ---------------- | ---------- | ----------------------------------------------- |
| `THOR_PROXY_URL` | (required) | Base URL of the proxy, e.g. `http://proxy:3001` |

## Architecture

```
OpenCode agent (cwd=/workspace/repos/acme-app)
  │
  ├─ git push ...                         → remote-cli.mjs git      → POST http://remote-cli:3004/exec/git    {args, cwd}
  ├─ gh pr create ...                     → remote-cli.mjs gh       → POST http://remote-cli:3004/exec/gh     {args, cwd}
  ├─ mcp                                  → remote-cli.mjs mcp      → POST http://proxy:3001/tools            {cwd}
  ├─ mcp slack                            → remote-cli.mjs mcp      → POST http://proxy:3001/slack/tools      {cwd}
  ├─ mcp slack postMessage '{"...":""}'  → remote-cli.mjs mcp      → POST http://proxy:3001/slack/tools/call  {name, arguments, cwd}
  ├─ approval status <id>                 → remote-cli.mjs approval → GET  http://proxy:3001/approval/<id>
  └─ approval list                        → remote-cli.mjs approval → GET  http://proxy:3001/approvals
```

The `mcp` wrapper reuses the same `remote-cli.mjs` pattern but targets the proxy instead of the remote-cli service. Each invocation is one HTTP request — no session, no SSE, no streaming transport. All requests include `cwd` in the JSON body; the proxy derives the repo name and validates access before proceeding.

### Config schema change

Add optional `proxies` array to `RepoConfigSchema`:

```typescript
const RepoConfigSchema = z.object({
  channels: z.array(z.string()).optional(),
  proxies: z.array(z.string()).optional(), // new — allowed upstream names; each must reference a key in the top-level `proxies` map
});
```

### Proxy changes needed

The proxy currently requires MCP session initialization (`initialize` → `initialized` handshake) before accepting `tools/list` or `tools/call`. For stateless CLI access we add **sessionless HTTP endpoints** (detailed contracts in Phase 1):

- Tools: `POST /tools`, `POST /:upstream/tools`, `POST /:upstream/tools/call`
- Approvals: `GET /approval/:id`, `GET /approvals`

All tool endpoints read `cwd` from the request body to scope access, returning 403 if the repo lacks the requested upstream in its `proxies` list.

**Removals** (deferred to Phase 3, while OpenCode still uses MCP): the `check_approval_status` synthetic tool, and the MCP session endpoints (`POST/GET/DELETE /:upstream`) replaced by the stateless endpoints above.

### OpenCode config change

Remove all `mcp` entries from `opencode.json`:

```json
{
  "model": "opencode/big-pickle",
  "permission": "allow",
  "mcp": {}
}
```

Tools become available via agent instructions (system prompt) that tell the agent to use the `mcp` CLI.

## Phases

### Phase 1 — Config schema + sessionless proxy endpoints

Extend the config schema and add plain HTTP endpoints to the proxy.

1. Add `proxies` field to `RepoConfigSchema` in `workspace-config.ts`, validating that entries reference keys in the top-level `proxies` map.
2. Add `POST /tools` — returns `{ upstreams: [{ name, description, toolCount }...] }` filtered by the repo's `proxies` list (repo derived from `cwd`).
3. Add `POST /:upstream/tools` — validates repo access (403 if denied), reads workspace config (live reload), connects to the upstream if needed, runs tool classification (allow/approve/hidden), returns `{ tools: [...] }` with descriptions.
4. Add `POST /:upstream/tools/call` — validates repo access, body `{ name, arguments }`, runs the same policy logic as the MCP `CallToolRequest` handler: `allow` forwards and returns the result, `approve` creates a pending approval and returns an action ID, `hidden` errors. Returns MCP `CallToolResult` JSON.
5. Add `GET /approval/:id` — searches all upstream stores by action ID, returns status + result if approved.
6. Add `GET /approvals` (optional, debugging) — pending approvals across all upstreams.
7. Add `tools`, `approval`, `approvals` to `RESERVED_PROXY_NAMES`, and register the static-path routes before the `/:upstream` wildcard.
8. Unit tests for the new endpoints. MCP session endpoints and `check_approval_status` stay until Phase 3.

**Exit criteria:**

- [ ] `proxies` field validated in config schema
- [ ] `curl -X POST http://localhost:3001/tools -d '{"cwd":"/workspace/repos/acme-app"}'` returns filtered upstream list
- [ ] `curl -X POST http://localhost:3001/slack/tools -d '{"cwd":"..."}'` returns tool list JSON
- [ ] `curl -X POST http://localhost:3001/slack/tools/call -d '{"name":"...","arguments":{},"cwd":"..."}'` executes a tool
- [ ] `curl http://localhost:3001/approval/<id>` returns approval status
- [ ] Request to an upstream not in repo's `proxies` list returns 403
- [ ] `tools`, `approval`, `approvals` are reserved proxy names
- [ ] New routes registered before `/:upstream` wildcard
- [ ] Existing MCP session endpoints still work (removed in Phase 3)
- [ ] `pnpm test` passes

### Phase 2 — CLI wrappers in OpenCode container

Add the `mcp` and `approval` CLI wrappers alongside existing `git`/`gh` wrappers.

1. Extend `remote-cli.mjs` to route the `mcp` subcommands to the proxy endpoints per the Architecture diagram (all requests carry `cwd`). `mcp <upstream> <tool> --help` fetches the tool list and prints the matched tool's description + schema.
2. Extend `remote-cli.mjs` with `approval` handling: `approval status <id>` → `GET /approval/<id>`, `approval list` → `GET /approvals`.
3. Add `docker/opencode/bin/mcp` and `docker/opencode/bin/approval` shell wrappers (like `git`/`gh`).
4. Set `THOR_PROXY_URL=http://proxy:3001` in docker-compose for the opencode service.
5. Manual test: `docker exec opencode mcp slack` returns tools.

**Exit criteria:**

- [ ] `mcp` prints available upstreams
- [ ] `mcp slack` prints tool list
- [ ] `mcp slack post_message --help` prints tool description and input schema
- [ ] `mcp slack post_message '{"channel":"C123","text":"hello"}'` returns CallToolResult JSON
- [ ] `approval status <id>` returns approval status
- [ ] `approval list` returns pending approvals
- [ ] `mcp` and `approval` wrappers are executable in OpenCode container
- [ ] No changes to existing `git`/`gh` wrappers

### Phase 3 — Remove MCP connections, inject tool list at session start

Disconnect OpenCode from direct MCP. The runner reads `config.json` to build tool instructions — no proxy dependency.

1. Remove `mcp` entries from `opencode.json`
2. Remove `check_approval_status` synthetic tool from MCP session handlers
3. Remove MCP session endpoints (`POST/GET/DELETE /:upstream` for MCP transport)
4. Update approval response text: `Run: approval status <id>` (replaces reference to removed `check_approval_status` tool)
5. In the runner (`packages/runner/src/index.ts`), before the first prompt to a new session: read workspace config (via `createConfigLoader`), look up the repo's `proxies` and each upstream's `allow`/`approve` tool lists, fetch tool descriptions (from `POST /:upstream/tools` or a cached config field), then build a tool instructions block and prepend it to the prompt (same injection point as pinned memory). Example injected block:

   ```
   [Available MCP tools — use the `mcp` CLI to call these]

   ## slack (allow)
   - post_message: Send a message to a Slack channel
   - read_thread: Read messages from a Slack thread

   ## atlassian (approve — requires human approval)
   - createJiraIssue: Create a new Jira issue

   Usage: mcp <upstream> <tool> '{"arg":"value"}'
   Run `mcp <upstream> <tool> --help` to see tool description and input schema.
   Run `approval status <id>` to check approval status.
   ```

   One section per upstream per classification (allow / approve), each tool listed with a one-line description.

6. Update static agent instructions in `docker/opencode/agents/build.md`: replace the hardcoded "MCP tools" section with a reference to the dynamic injection, and document the `mcp` CLI usage pattern.
7. Verify OpenCode starts without MCP connections.
8. Update the `docs/feat/mvp.md` architecture diagram.

**Exit criteria:**

- [ ] `opencode.json` has empty or no `mcp` block
- [ ] MCP session endpoints removed (`POST/GET/DELETE /:upstream` for MCP transport)
- [ ] `check_approval_status` synthetic tool removed
- [ ] Approval response text says `Run: approval status <id>`
- [ ] New sessions receive tool list with descriptions built from config.json
- [ ] Tool list shows allow/approve classification per upstream
- [ ] Tool list reflects the repo's configured upstreams (not all upstreams)
- [ ] Agent instructions reference `mcp` CLI and `--help` for schema discovery
- [ ] OpenCode container starts cleanly with no MCP connections
- [ ] `mvp.md` reflects the new architecture
- [ ] `pnpm test` passes

### Phase 4 — Error handling and edge cases

Make the CLI helpful when things go wrong: fuzzy matching, auto-suggestions, and clear error messages. On any failure the CLI writes to stderr and exits 1 (exceptions noted below).

**Fuzzy matching** — unknown upstream or tool names are matched (Levenshtein or substring) against the configured set; the message names the closest match and lists the alternatives so the agent can self-correct in one retry.

**Schema errors** — the upstream MCP server validates arguments and returns the error in CallToolResult (no client-side validation). The CLI appends the tool's input schema as a `[hint]` so the agent can fix the call without a separate `--help` round-trip.

**Approval flow** — an approve-gated call (e.g. `mcp atlassian createJiraIssue '{...}'`) returns an explicit JSON result (`type: "approval_required"`, `actionId`, `proxyName`, `tool`, status command) on stdout; the runner validates that shape and emits an `approval_required` progress event. `approval status <id>` (separate CLI) hits `GET /approval/:id`, which searches all upstream stores and returns pending/approved/rejected + result.

| Case                       | Trigger                           | Behavior                                                                |
| -------------------------- | --------------------------------- | ----------------------------------------------------------------------- |
| Unknown upstream           | `mcp slck`                        | Suggest closest, list available upstreams                               |
| Unknown tool               | `mcp slack post_msg`              | Suggest closest, list that upstream's tools                             |
| Schema error               | upstream rejects args             | Raw CallToolResult + `[hint]` input schema                              |
| Proxy unreachable          | proxy down                        | `Failed to reach proxy...`                                              |
| Upstream unreachable       | proxy up, upstream down (502)     | `Upstream "slack" is unreachable...`                                    |
| Upstream timeout           | call exceeds 120s                 | `Timeout calling "<tool>" on "<upstream>"...`                           |
| Bad cwd                    | cwd not under `/workspace/repos/` | `Cannot determine repo from cwd...`                                     |
| Repo not in config         | cwd resolves to unknown repo      | `Repo "..." not found. Available: ...`                                  |
| Repo has no `proxies`      | repo exists, none configured      | `No upstreams configured for repo "..."` — exit 0 (empty, not an error) |
| Config changed mid-session | tool removed after discovery      | "Unknown tool" + fuzzy suggestion; agent re-runs `mcp` to refresh       |

**Exit criteria:**

- [ ] `mcp slck` suggests "slack" via fuzzy match
- [ ] `mcp slack post_msg` suggests "post_message" via fuzzy match
- [ ] Schema errors auto-include the tool's input schema as a hint
- [ ] `approval status <id>` returns approval status from proxy
- [ ] Approval flow produces same JSON signals runner already parses
- [ ] All error cases output helpful messages to stderr with exit 1
- [ ] `pnpm test` passes

## Out of scope

- MCP stdio transport bridge (may add later if needed)
- Streaming tool responses (all calls are request/response for now)
- Auth/token on the CLI endpoints (proxy is internal, same trust boundary as existing MCP)

## Decision Log

| #   | Decision                                                                | Rationale                                                                                                                                                                                                                          |
| --- | ----------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 1   | Plain CLI over MCP stdio transport                                      | No session state, same pattern as git/gh wrappers. MCP stdio adds complexity for no benefit when the goal is stateless access.                                                                                                     |
| 2   | CLI name: `mcp`                                                         | Short, accurate, no conflicts inside the controlled container PATH.                                                                                                                                                                |
| 3   | Upstream and tool names as subcommands                                  | `mcp slack postMessage '{}'` reads naturally — arity determines the action (0 args = upstreams, 1 = tools, 2+ = call). No `list`/`call` verbs.                                                                                     |
| 4   | JSON string as single tool argument                                     | Handles any schema (nested objects, arrays) without flag-to-JSON mapping. One input format, no ambiguity.                                                                                                                          |
| 5   | Sessionless HTTP endpoints on proxy                                     | Avoids MCP handshake overhead — one request, one response. Proxy reuses existing upstream connections internally.                                                                                                                  |
| 6   | Extend `remote-cli.mjs` instead of new script                           | DRY — same HTTP client, error handling, and container PATH setup.                                                                                                                                                                  |
| 7   | Raw MCP CallToolResult JSON output                                      | Agent parses JSON natively; no transformation or pretty-printing.                                                                                                                                                                  |
| 8   | Remove MCP session endpoints and synthetic `check_approval_status` tool | No MCP session clients remain after migration; removing dead code reduces surface area.                                                                                                                                            |
| 9   | Per-repo upstream scoping in config.json                                | Replaces per-repo `opencode.json` MCP blocks with centralized config. Definitions stay DRY, proxy enforces access server-side.                                                                                                     |
| 10  | Repo identity from `cwd`                                                | CLI already sends `cwd`; proxy extracts repo name from path (`/workspace/repos/<repo>`). No extra env var or header.                                                                                                               |
| 11  | Runner reads config.json for tool list, not proxy                       | No proxy dependency at session start. Agent uses `mcp <upstream> <tool> --help` to discover schema on demand.                                                                                                                      |
| 12  | Fuzzy match on unknown upstream/tool                                    | Agent typos are common; suggesting the closest match + alternatives saves a round-trip.                                                                                                                                            |
| 13  | Auto-include schema on tool call errors                                 | Appending the input schema as a hint lets the agent self-correct in one retry instead of two (error → --help → retry).                                                                                                             |
| 14  | Separate `approval` CLI from `mcp` CLI                                  | Approvals are a different workflow (async, cross-upstream, involves Slack/gateway). Keeps each CLI single-purpose.                                                                                                                 |
| 15  | `approval status` searches across all upstream stores                   | Action IDs are globally unique; the agent shouldn't need to remember which upstream an approval came from.                                                                                                                         |
| 16  | Stateless CLI over auto-reconnect                                       | Auto-reconnect requires forking the MCP SDK transport or patching OpenCode internals. A conscious, reversible bet on current MCP spec immaturity: if OpenCode gains native session recovery, re-evaluate and retire the CLI layer. |
| 17  | Defer MCP endpoint removal to Phase 3                                   | Removing session endpoints while OpenCode still uses MCP creates an outage. Keep old and new endpoints running until Phase 3 cuts over.                                                                                            |
| 18  | Include tool descriptions in injected prompt                            | Tool names alone cause the agent to hallucinate argument schemas. Include one-line descriptions per tool.                                                                                                                          |
| 19  | Keep per-upstream approval resolve endpoint                             | Gateway calls `POST /:upstream/approval/:id/resolve`; only the CLI read path gets cross-upstream search. Stays per-upstream until gateway is updated.                                                                              |
| 20  | Explicit JSON for aliases and approvals (not hidden metadata)           | Alias registration happens at the producer via `aliases.jsonl`; approval-required responses are explicit JSON CLI output. The runner no longer parses hidden metadata from stdout/stderr.                                          |
