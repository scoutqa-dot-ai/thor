# Plan: GitHub MCP Integration

**Date**: 2026-03-11
**Goal**: Add GitHub MCP server and git-mcp server to Thor via the existing proxy.

## Phases

### Phase 1 — GitHub MCP proxy config + docker-compose wiring ✅

- Create `proxy.github.json` with upstream URL and allowlisted tools (read-only to start)
- Add port 3013 to proxy healthcheck
- Add `GITHUB_PAT` env var to proxy service
- Add `github` MCP entry to `docker/opencode/opencode.json`
- Remove all host-exposed ports from internal services — only ingress exposes a port

### Phase 2 — Test GitHub MCP connectivity ✅

- Proxy connects to upstream, lists 23 tools, exposes 12 via allowlist
- `get_file_contents` successfully reads files from public repos
- All 8 proxy test assertions pass

### Phase 3 — Git MCP server for credential isolation

- Create `@thor/git-mcp` package with a single `git` tool that executes `git` commands
- PAT lives only in the git-mcp container; injected via `GIT_ASKPASS` at exec time
- OpenCode container has no git credentials — it calls the `git` tool through the proxy
- Wire into docker-compose as a new service on port 3004
- Proxy on port 3014 (`proxy.git.json`) forwards to git-mcp
- Shared `/workspace` volume so worktrees created by git-mcp are visible to OpenCode

**Exit criteria**: git-mcp starts, proxy lists the `git` tool, a `git status` call works.

## Decision Log

| #   | Decision                                                                    | Rationale                                                                                   |
| --- | --------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------- |
| 1   | Use remote hosted MCP (`api.githubcopilot.com/mcp/`) not local Docker image | Zero maintenance, same tools, simpler deployment                                            |
| 2   | Classic PAT for initial setup                                               | Works with org repos without org admin opt-in for fine-grained tokens                       |
| 3   | Read-only allowlist to start                                                | Least privilege; add write tools later with approval gates                                  |
| 4   | Use `/readonly` URL suffix                                                  | Double safety — even if allowlist drifts, upstream restricts to reads                       |
| 5   | Remove host port mappings from internal services                            | Proxy has no auth — exposing ports leaks unauthenticated MCP endpoints to the host network  |
| 6   | Git MCP with single `git` exec tool instead of per-operation tools          | Full git flexibility; server-side rulesets handle protection, MCP just isolates credentials |
| 7   | No client-side blocklist for git commands                                   | GitHub rulesets already configured server-side; git-mcp's role is credential isolation only |

## Disallowed GitHub MCP Tools

Tools available in the upstream `readonly` toolset but excluded from the allowlist:

| Tool                  | Reason                                                                              |
| --------------------- | ----------------------------------------------------------------------------------- |
| `search_repositories` | Discovery beyond known repos is unnecessary; agent operates on a fixed set of repos |
| `search_users`        | No use case; could leak org member info                                             |
| `get_label`           | Low value; labels are visible via issue/PR reads                                    |
| `get_latest_release`  | Not needed for current workflows                                                    |
| `get_release_by_tag`  | Not needed for current workflows                                                    |
| `get_tag`             | Not needed for current workflows                                                    |
| `get_team_members`    | Sensitive org structure info; no current use case                                   |
| `get_teams`           | Sensitive org structure info; no current use case                                   |
| `list_issue_types`    | Low value metadata                                                                  |
| `list_releases`       | Not needed for current workflows                                                    |
| `list_tags`           | Not needed for current workflows                                                    |

## Out of Scope
