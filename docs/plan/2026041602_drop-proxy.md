# Drop `@thor/proxy` ã Consolidate into `@thor/remote-cli`

> **Partially superseded:** The `getRepoProxies` ã `getRepoUpstreams` rename described
> here no longer matters ã the function and the `repos[*].proxies` field it read have
> been removed. The proxy consolidation itself stands; only the per-repo allowlist
> piece is gone. See current `packages/remote-cli/src/mcp-handler.ts` and
> `packages/common/src/proxies.ts`.

**Goal**: Eliminate `packages/proxy` as a separate service. Move all MCP-proxy logic (upstream connections, tool listing/calling, policy enforcement, approval workflow) into `packages/remote-cli` as `/exec/mcp` and `/exec/approval` endpoints. One fewer container, one fewer network hop, consistent API.

## Motivation

- The proxy and remote-cli are both Express HTTP servers with nearly identical patterns (JSON request/response, `{ stdout, stderr, exitCode }` shape, Thor tracing headers, server-side policy).
- The proxy's only unique capability is acting as an MCP client to upstream servers. This is a library concern (`@modelcontextprotocol/sdk`), not an architectural one.
- Merging reduces:
  - 1 Docker container + health check
  - 1 port allocation (3001)
  - Network hops for tool calls
  - Config surface (`proxyPort`, `proxyHost`, `THOR_PROXY_URL`)
- Unifying to `/exec/mcp` pattern makes the API consistent: git, gh, scoutqa, langfuse, mcp all use `POST /exec/{tool}`.

## Scope

**In scope:**

- Move MCP upstream connection logic into remote-cli
- Move policy classification + approval workflow into remote-cli
- Expose as `POST /exec/mcp` and `POST /exec/approval` (unified API pattern)
- Delete `proxy-cli.ts` (remote-cli.mjs already dispatches to `/exec/{endpoint}`)
- Update gateway approval resolution to hit remote-cli
- Update docker-compose, Dockerfile, runner config
- Delete `packages/proxy`
- Rename `getRepoProxies` ã `getRepoUpstreams` in common (in blast radius)
- Add `RESOLVE_SECRET` gating on approval resolution endpoint (security hardening)

**Out of scope:**

- Changing the MCP protocol or approval UX
- Changing the policy model (allow/approve/hidden stays)
- Refactoring remote-cli's existing endpoints (git/gh/scoutqa/langfuse)

### API Design

MCP endpoint (`POST /exec/mcp`):

```
Body: { args: string[] }

`/exec/mcp` does not accept `cwd` or `directory` (agent-supplied directory signals
are not used for MCP routing ã see profile-routing Decision 21). Git/gh/sandbox
endpoints still use `cwd`.

args = []                                          ã list upstreams
args = ["<upstream>"]                              ã list tools
args = ["<upstream>", "<tool>", "--help"]           ã tool schema
args = ["<upstream>", "<tool>", "<json-args>"]      ã call tool

Response: { stdout, stderr, exitCode }
```

Approval endpoint (`POST /exec/approval`):

```
Body: { args: string[] }

args = ["status", "<id>"]   ã check approval status
args = ["list"]              ã list pending approvals

Response: { stdout, stderr, exitCode }
```

Resolve via secret-gated `/exec/mcp`:

```
POST /exec/mcp
Headers: { x-thor-resolve-secret: string }  ã must match RESOLVE_SECRET env var
Body: { args: ["resolve", "<actionId>", "<decision>", "<reviewer>", "<reason>?"] }

WITHOUT secret header: ã { stderr: "Unknown subcommand: resolve", exitCode: 1 }
WITH secret header:    ã lookup approval by actionId ã execute stored tool+args ã return result

Response: { stdout, stderr, exitCode }
```

Same endpoint, secret as auth. The agent's `mcp` CLI wrapper never sends the secret header, so `mcp resolve ...` from the agent always fails. Gateway sends the secret, so its resolve calls succeed. No new endpoints needed.

## Phases

### Phase 1 ã Move MCP proxy logic into remote-cli

**Files to create/modify:**

| File                                        | Action          | Notes                                                                                                                      |
| ------------------------------------------- | --------------- | -------------------------------------------------------------------------------------------------------------------------- |
| `packages/remote-cli/src/upstream.ts`       | Copy from proxy | MCP client connection + reconnect logic                                                                                    |
| `packages/remote-cli/src/policy-mcp.ts`     | Copy from proxy | `classifyTool`, `validatePolicy`, error classes                                                                            |
| `packages/remote-cli/src/approval-store.ts` | Copy from proxy | Filesystem approval persistence                                                                                            |
| `packages/remote-cli/src/unwrap-result.ts`  | Copy from proxy | MCP result formatting                                                                                                      |
| `packages/remote-cli/src/mcp-handler.ts`    | New             | Args parser + handler for `/exec/mcp` and `/exec/approval`. Mirrors the command parsing from proxy-cli.ts but server-side. |
| `packages/remote-cli/src/index.ts`          | Edit            | Add `/exec/mcp` and `/exec/approval` routes, add startup logic for eager upstream connect, update health endpoint          |
| `packages/remote-cli/package.json`          | Edit            | Add `@modelcontextprotocol/sdk`, `zod` deps                                                                                |

**No tsup config change needed.** Root `tsup.config.ts` already handles this: `noExternal` only inlines `@thor/*` workspace packages. The MCP SDK stays in `node_modules` and is not bundled.

**Startup sequence:** CLI endpoints (`/exec/git`, `/exec/gh`, etc.) available at T+0. MCP upstreams connect asynchronously in the background. Health endpoint always returns 200 if Express is up. Response body includes MCP upstream info for debugging. Docker compose healthcheck uses this simple alive check.

**MCP logger:** Use `createLogger("mcp")` for MCP-related log events, separate from the existing `createLogger("remote-cli")`.

**CWD validation for MCP:** MCP endpoints do not accept `cwd` or `directory` for authz/profile routing (see profile-routing Decision 21). `/exec/approval` does not require cwd either ã approvals are not repo-scoped.

**Approval store path:** Use absolute path `/workspace/data/approvals` (not relative). The proxy uses `data/approvals` relative to its WORKDIR (`/workspace`). Remote-cli has a different WORKDIR. Use an absolute path to avoid silent data loss.

**Exit criteria:**

- `GET /health` returns 200 with MCP upstream info in body
- `POST /exec/mcp` with `{args: []}` returns upstream list
- `POST /exec/mcp` with `{args: ["slack", "postMessage", '{"...":"..."}']}` calls tool
- `POST /exec/approval` with `{args: ["status", "uuid"]}` returns approval
- `POST /exec/mcp` with `{args: ["resolve", ...]}` + secret header resolves approval
- `POST /exec/mcp` with `{args: ["resolve", ...]}` WITHOUT secret returns error
- All existing remote-cli endpoints (git/gh/scoutqa/langfuse) unaffected
- Proxy tests adapted and passing against remote-cli

## Decision Log

| #   | Decision                                                | Reason                                                                                                                                                                                             |
| --- | ------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 1   | Use `/exec/mcp` and `/exec/approval` endpoint pattern   | Consistent with all other remote-cli endpoints. Eliminates route conflicts.                                                                                                                        |
| 2   | Delete proxy-cli.ts entirely (no merge needed)          | remote-cli.mjs already dispatches `POST /exec/{endpoint}` for any endpoint name. Zero wrapper code changes.                                                                                        |
| 3   | Copy source files rather than creating a shared library | Only one consumer (remote-cli). If a second appears, extraction is trivial.                                                                                                                        |
| 4   | Server-side MCP command parsing (in mcp-handler.ts)     | Mirrors how langfuse/scoutqa work. Command parsing that was in proxy-cli.ts moves to server.                                                                                                       |
| 5   | Remote-cli port stays 3004                              | No reason to change; avoids config churn.                                                                                                                                                          |
| 6   | Accept credential consolidation trade-off               | Internal tooling; the agent already has transitive access to both services, and remote-cli already runs arbitrary shell commands. MCP read-only API keys don't materially change the blast radius. |
| 7   | Keep v1 approval button format for now                  | Remove in a separate PR once safe.                                                                                                                                                                 |
| 8   | Rename `getRepoProxies` ã `getRepoUpstreams`            | "Proxy" naming would be stale; the rename is small and in blast radius.                                                                                                                            |
| 9   | No tsup config change needed                            | Root tsup config already externals all non-@thor packages. MCP SDK won't be bundled.                                                                                                               |
| 10  | Secret-gate resolve on `/exec/mcp`                      | Addresses self-approval risk: the secret header on the same endpoint prevents agent access, with no new endpoint.                                                                                  |
| 11  | Use absolute approval store path                        | The proxy and remote-cli WORKDIRs may differ; use the absolute `/workspace/data/approvals` to avoid silent data loss.                                                                              |
| 12  | Extract a testable `createRemoteCliApp()` boot path     | Lets Phase 1 verify `/exec/mcp` and `/exec/approval` via HTTP without starting the production listener during tests.                                                                               |
| 13  | Drop legacy approval fallback paths and button format   | Project is greenfield. Per-upstream approval records are simpler and avoid guessing the upstream at resolve time.                                                                                  |

## Risks

| Risk                                              | Mitigation                                                                       |
| ------------------------------------------------- | -------------------------------------------------------------------------------- |
| Remote-cli container needs MCP SDK + larger image | Marginal size increase (~2MB); already has express                               |
| Startup time increases (eager upstream connect)   | Upstream connect is async; CLI endpoints remain available at T+0                 |
| Breaking in-flight approvals during deploy        | Approval data on shared volume. Deploy during low-activity window.               |
| Credential consolidation                          | Documented trade-off (Decision #6). All internal, same Docker network.           |
| Self-approval via resolve subcommand              | Mitigated: secret-gated on /exec/mcp (Decision #10). Agent never has the secret. |
