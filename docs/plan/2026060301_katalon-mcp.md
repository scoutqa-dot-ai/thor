# Katalon True Platform remote MCP (read-only, profile-aware)

Add Katalon True Platform's hosted **remote MCP server** as a new proxy upstream so it joins
`atlassian` / `grafana` / `langfuse` / `posthog` in the profile-aware MCP proxy registry and is
reachable through the existing `mcp` CLI. Authentication is **exactly one method**: a Katalon
**API key** (`KATALON_API_KEY`) sent as HTTP Basic auth. The allow list exposes **read-only
tools only**; every create/update/delete/move/manage/schedule tool stays `hidden`.

## Why

Katalon True Platform ships a hosted remote MCP server (`https://<sub-domain>.katalon.io/mcp`,
streamable HTTP) that exposes test cases, suites, runs, requirements, defects, executions, and
analytics. Adding it to `resolveProxyConfig` reuses the credential-routing path that already
gives PostHog/Atlassian/Langfuse profile suffixes (`*_<PROFILE>` → global fallback), so we get
profile routing and the existing `mcp` CLI surface for free — no bespoke `/exec` endpoint.

The published docs only describe an interactive OAuth browser-login flow, but the server also
accepts **HTTP Basic auth with the API key as the credential**. This was verified live against
`prodgen3platform.katalon.io/mcp`: an `initialize` handshake with
`Authorization: Basic base64("<arbitrary-username>:<api-key>")` returns a successful `result`
(server `TestOps/Katalon True Platform Protected MCP Server` v2.14.5) with no browser prompt.
The username component is **ignored** by the server; the API key is the password.

## Scope

**In scope**

- Add `katalon` to `PROXY_NAMES` (inserted alphabetically: `atlassian`, `grafana`, `katalon`,
  `langfuse`, `posthog`) with a **read-only** allow list and an empty approve list.
- Resolve the Katalon upstream in `resolveProxyConfig`: `KATALON_API_KEY` + `KATALON_BASE_URL`
  are one strict bundle (profile suffix first, global fallback, fail hard on a partial bundle,
  per profile-routing Decision 12 / Langfuse Decision 3). Base URL normalized via the shared
  `envBaseUrl` helper (trailing slash stripped); upstream URL = `${base}/mcp`. Auth header =
  `Basic base64("${KATALON_BASIC_USER}:${KATALON_API_KEY}")`.
- Update `proxies.test.ts`: `PROXY_NAMES` order assertion, katalon profile/global/partial-bundle
  resolution cases, and the per-upstream allow/approve disjointness case.
- Env discipline surfaces: `.env.example`, `README.md` Deployment Configuration table,
  GitHub workflow env blocks (`core-e2e.yml`, `sandbox-e2e.yml`) + the `.env` injection step,
  and `security-model.md`. `docker-compose.yml` passes MCP creds through `env_file: .env`, so it
  needs a change only if Katalon vars must be enumerated there (confirm in Phase 4).

**Out of scope**

- Katalon **write** tools (create/update/delete/move/duplicate/manage/schedule/rerun, defect
  creation, manual-run/AI-session creation). They are excluded from the allow list → classify as
  `hidden` and are unreachable. No approve/write path until a concrete use case appears (keeps
  `APPROVAL_TOOL_NAMES` assertion unchanged).
- **Non-mutating workflow-scaffolding tools** (`build_run_configuration`, `build_schedule`,
  `build_private_local_testing_config`, `get_upload_test_cloud_app_url`,
  `prepare_test_cloud_tunnel_start`). These do not mutate server state, but exist only to stage
  a subsequent write (scheduling a run, uploading an app, starting a tunnel) and one
  (`prepare_test_cloud_tunnel_start`) returns API keys in its payload. Excluded from the
  read-only surface; revisit if a read-only consumer needs them.
- A second auth method. **Only `KATALON_API_KEY`** is supported — no OAuth browser flow, no
  bearer-token variant, no email/password. (Verified: bearer was rejected; Basic-with-API-key
  works.)
- A `grafana-mcp`-style sidecar — Katalon MCP is hosted remote like Atlassian/PostHog, so it
  connects directly with no new compose service.
- Per-profile tool allow/approve policy (stays global per integration).
- A dedicated `katalon` skill or prompt wrapper. The generic `mcp katalon <tool>` surface is
  sufficient; add a skill later only if a guided workflow is needed.

## Environment resolution contract

For a profile `LABS`, the **bundle** is both `KATALON_API_KEY` and `KATALON_BASE_URL` — strict
(both scoped, or both global):

- `KATALON_API_KEY_LABS` + `KATALON_BASE_URL_LABS`
- else the global `KATALON_API_KEY` + `KATALON_BASE_URL`
- else disabled.
- If exactly one of the two `*_LABS` vars is set → throw, naming the missing leg (no silent mix
  of profile and global scope; a scoped key must route at the tenant's own URL).

The API key and the tenant subdomain are tied together — a key issued for tenant A will not
authenticate against tenant B's URL — so they form one strict bundle exactly like Langfuse's
`pk`/`sk`/host. There is no default URL: if no bundle resolves, Katalon is disabled. The scheme
is **not enforced** (http accepted for any self-hosted/internal deployment), matching the
Langfuse precedent; `envBaseUrl` strips trailing slashes to avoid `${base}//mcp`.

Upstream: `url = ${envBaseUrl(base)}/mcp`, `headers.Authorization = Basic base64(user:apiKey)`,
where `user` is a fixed module constant (`KATALON_BASIC_USER`). The server ignores the username,
so any stable non-empty value works; we send a constant rather than leak the key into the
username position.

## Decision log

| #   | Decision                                                                                                                                    | Rationale                                                                                                                                                                                                        | Rejected                                                                                                                                 |
| --- | ------------------------------------------------------------------------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------- |
| 1   | Katalon is an MCP proxy upstream, not a CLI                                                                                                 | Hosted remote MCP exists; reusing `resolveProxyConfig` gets profile routing + the `mcp` CLI for free and avoids a bespoke `/exec/katalon` surface                                                                | A dedicated `/exec/katalon` CLI integration                                                                                              |
| 2   | Support exactly one auth method: `KATALON_API_KEY` via HTTP Basic                                                                           | Verified live: Basic auth with the API key authenticates with no browser prompt; bearer was rejected. A single credential keeps the resolver simple and the operator contract unambiguous                        | OAuth browser flow (non-headless, can't run on the server); bearer token (rejected by server); email+password (extra secret, no benefit) |
| 3   | API key goes in the Basic **password** position, fixed constant username (`KATALON_BASIC_USER`)                                             | Verified working form is `base64("<arbitrary-user>:<api-key>")` — the server ignores the username. Sending a constant avoids leaking the key into the username position and keeps the header deterministic       | Key as username with blank password (untested against this endpoint); empty username (untested)                                          |
| 4   | Treat `KATALON_API_KEY` + `KATALON_BASE_URL` as one strict bundle (both scoped, or both global)                                             | The key is tenant-specific and only valid against its own subdomain URL; a half-scoped bundle is almost certainly an operator typo and would route a scoped key at the wrong tenant. Mirrors Langfuse Decision 3 | Independent per-var fallback (could pair a profile key with the global URL → auth failure against the wrong tenant)                      |
| 5   | Routing var is `KATALON_BASE_URL` (full base, e.g. `https://prodgen3platform.katalon.io`), normalized via `envBaseUrl`, URL = `${base}/mcp` | Matches the repo-wide `envBaseUrl` convention (Langfuse), supports regional/self-hosted hosts, and keeps the domain out of code                                                                                  | `KATALON_SUBDOMAIN` + hardcoded `.katalon.io/mcp` (bakes the domain into code, breaks on any non-`katalon.io` host)                      |
| 6   | Read-only allow list only; `KATALON_APPROVE = []`                                                                                           | Goal is a read-only Katalon surface; no write use case today; keeps the `APPROVED_PROXY_TOOLS` vs `APPROVAL_TOOL_NAMES` assertion unchanged                                                                      | Expose write tools behind approvals now (adds disclaimer-schema + approval-event work with no current need)                              |
| 7   | Exclude non-mutating `build_*` / `prepare_*` / `get_upload_*` scaffolding tools                                                             | They exist only to stage writes (schedule/upload/tunnel) and one returns API keys in its payload; out of a strictly read-only surface                                                                            | Allow them as "reads" (they are write-workflow steps, and one leaks credentials)                                                         |
| 8   | Insert `katalon` alphabetically in `PROXY_NAMES`                                                                                            | Stable, predictable ordering in listings/health; only the one order assertion in `proxies.test.ts` changes                                                                                                       | Append at end                                                                                                                            |

## Phases

### Phase 1 — Registry + credential routing (`@thor/common`)

In `packages/common/src/proxies.ts`: add `katalon` to `PROXY_NAMES` (alphabetical), define the
read-only `KATALON_ALLOW` and `const KATALON_APPROVE: string[] = []`, add the `KATALON_BASIC_USER`
constant, and add the `resolveProxyConfig` strict-bundle branch (`KATALON_API_KEY` +
`KATALON_BASE_URL`, profile-first with global fallback, throw on partial profile bundle, URL =
`${envBaseUrl(base)}/mcp`, Basic auth header). Update `proxies.test.ts`: order assertion,
katalon profile/global/partial cases, and the allow/approve disjointness case.

**Exit:** `pnpm --filter @thor/common test` green; katalon resolves profile-first with global
fallback, throws on a partial profile bundle, and is absent when neither var is set.

### Phase 2 — Policy validation sanity (`@thor/remote-cli`)

No code change expected (`validatePolicy` / `classifyTool` are generic). Confirm the existing
policy tests still pass and that the read-only allow list contains no tool absent from the live
upstream (no policy drift) and no allow/approve overlap.

**Exit:** `pnpm --filter @thor/remote-cli test` green; connecting katalon logs `upstream_ready`
with the expected allow count and no `policy_drift` / `PolicyOverlapError`.

### Phase 3 — Env discipline surfaces

Update `.env.example` (`KATALON_API_KEY` + `KATALON_BASE_URL` with a profile-scoped example and
a note that both form one bundle), `README.md` Deployment Configuration table (two rows,
Service = remote-cli, profile-variant naming), `security-model.md`, and confirm whether
`docker-compose.yml` needs explicit entries or relies on `env_file: .env`.

**Exit:** every required surface documents the two vars consistently; no var documented only in
code.

### Phase 4 — CI env wiring

Add `KATALON_API_KEY` / `KATALON_BASE_URL` to the env blocks of `.github/workflows/core-e2e.yml`
and `sandbox-e2e.yml` and to their `.env` injection step, using repo secrets/vars (or a
CI-fake/skip pattern consistent with how an optional MCP upstream is treated when creds are
absent — katalon simply does not appear in `mcp` listings without creds).

**Exit:** workflows reference the vars; absence of creds disables katalon without failing the run.

### Phase 5 — Integration verification

Push the branch, let core/sandbox e2e run, and verify live: `mcp katalon` lists exactly the
read-only tools; a write tool (e.g. `create_test_case`) is `hidden`/unreachable; a read tool
(e.g. `list_projects`) returns data over Basic auth. Open a PR against `main` once green.

**Exit:** read tools reachable, write tools hidden, katalon present only when creds resolve;
required push checks green.
